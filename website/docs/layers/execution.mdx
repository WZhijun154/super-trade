# Execution — `super_trade.execution`

The live-trading layer. It runs a strategy against a **broker**, with risk
controls, and is the bridge between backtest and live: the **same**
`Strategy` drives a simulated broker (dry-run / backtest) or a QMT broker (live on
a **simulation account**).

:::danger[Sensitive layer]
This layer can place orders. It defaults to **`dry_run=True`** (logs orders,
sends nothing), and is intended for a **QMT 模拟账户 (simulation account)**. Keep the
`RiskManager` limits on, and only set `dry_run=False` once you've watched the
dry-run output.
:::

A `Broker` exposes `cash()`, `positions()`, and `place_order()`. `SimBroker` fills
against an in-memory `Account` (cash + positions in real 手/shares), reusing the
backtest `CostModel` so costs match. `QmtBroker` submits via `xtquant.xttrader`.

## The scan loop

`ExecutionEngine.scan_once()` is one sweep of the universe:

1. read recent bars per symbol → evaluate the strategy's target weight + price;
2. mark the account to market; apply the **daily-loss halt**;
3. **stop-loss exits first** (a name just stopped out is not re-bought this scan);
4. **signal entries/exits**, sized by the `RiskManager`;
5. submit orders through the broker (or just log them, in dry-run).

```python
from super_trade.execution import SimBroker, RiskManager, ExecutionEngine
from super_trade.backtest import SmaCross

engine = ExecutionEngine(
    broker=SimBroker(cash=1_000_000),
    store=store, strategy=SmaCross(10, 30), risk=RiskManager(),
    universe=["600519", "000001"], dry_run=True,
)
engine.scan_once()              # one pass
engine.run(interval_seconds=10) # scan every 10s (blocks; Ctrl-C to stop)
```

### Target-weight rebalancing

A strategy's `signal` is a **target weight** — the *fraction of equity* to hold in a
name, not a 0/1 in-or-out flag. Each bar the event-driven engine sizes the desired
position (`weight × equity / price`, clamped to the per-name cap, rounded to whole
lots) and trades only the **difference** vs what it already holds. So a *changing*
target **scales a position in and out over many bars** — the same name is bought and
sold repeatedly — instead of flipping fully on/off.

`ScaledRsiReversion` shows this: it ramps the weight from 0 (RSI ≥ high) to 1.0
(RSI ≤ low), so the engine accumulates as the name gets more oversold and trims as it
recovers. The ±half-lot rounding is a deadband that keeps a steady target from
churning a lot back and forth each bar.

## Risk management

`RiskManager` is the **safety layer between a strategy's signal and an actual
order**. The strategy says *whether* to be long; the `RiskManager` decides *how much*
you may buy and *whether to block or force* a trade — so risk rules live in one
place. It's used by the event-driven backtest and live execution (not the
vectorized engine).

### Limits (`RiskLimits`)

| Limit | Default | Meaning |
|---|---|---|
| `max_position_weight` | 0.20 | at most 20% of equity in any one name |
| `max_gross_exposure` | 0.95 | at most 95% invested in total (keep a cash buffer) |
| `max_daily_loss` | 0.05 | halt new buys once down 5% on the day |
| `stop_loss` | 0.08 | exit a name down 8% from its entry price |
| `max_orders_per_day` | 200 | cap on orders per session |

### What it does

- **Position sizing** — `max_buy_shares(symbol, price, equity, broker)` returns the
  largest **board-lot-aligned** buy allowed: the **minimum** of three budgets, floored
  to whole 手 (`shares = (budget // (price × 100)) × 100`):
  - *per-name*: `max_position_weight × equity` − what you already hold,
  - *gross*: `max_gross_exposure × equity` − total invested,
  - your available **cash**.
- **Stop-loss** — `stop_triggered(pos, price)` is true when
  `price ≤ avg_price × (1 − stop_loss)`; the engine then sells that name.
- **Daily-loss halt** — `start_day(equity)` snapshots the morning equity;
  `update_daily_loss(equity)` halts once equity drops past `max_daily_loss`.
- **Order-count cap** — `can_trade()` becomes false after `max_orders_per_day`.
- **Kill switch** — `kill()` halts everything immediately.

### Buys are gated; exits never are

When halted (daily-loss hit, kill switch, or order cap), **new buys are blocked but
sells and stop-losses always go through** — you can always reduce risk. (`can_trade()`
is checked before entries; exits bypass it.)

### Per-session state

A `RiskManager` carries intraday state (the day's starting equity, order count, halt
flag), so **create a fresh one per backtest/session** and call `start_day()` at the
open — the event-driven backtest and `ExecutionEngine` do this for you.

```python
from super_trade.execution import RiskManager, RiskLimits

risk = RiskManager(RiskLimits(max_position_weight=0.15, stop_loss=0.06))
```

## A-share market realism (`MarketRules`)

The `EventDrivenBacktest` doesn't just model your *account* (cash, lots, stops) — it
also models the *market*, so an order that wouldn't have filled in reality doesn't
fill in the backtest. These rules are on by default via `MarketRules` and apply in
the bar loop (the `SimBroker` stays a pure fill engine):

- **T+1** — shares bought today can't be sold until the next trading day. A same-day
  buy is unsellable even by a stop-loss; the position is trapped until tomorrow.
- **涨跌停 (price limits)** — you can't BUY at the upper limit (no sellers) or SELL at
  the lower limit (no buyers). The band is inferred from the code prefix via
  `limit_pct`: STAR/ChiNext (688/300) ±20%, Beijing (4_/8_/920) ±30%, main boards
  ±10% (ST's true ±5% isn't detectable from the code → falls back to ±10%).
- **停牌 (suspension)** — a bar with `volume == 0` (or no bar at all) doesn't trade.
- **Partial fills** — a single order takes at most `participation_rate` (default
  **10%**) of the bar's volume; the rest doesn't fill.

```python
from super_trade.execution import EventDrivenBacktest, MarketRules

# defaults are realistic; relax any rule explicitly
rules = MarketRules(participation_rate=0.25, t_plus_1=False)
EventDrivenBacktest(store, strategy, universe=["600519"], rules=rules).run()
```

**Costs** were also brought current: stamp tax **5 bp** (sell-only, the rate since
2023-08-28), a **¥5 per-order commission floor**, and the **过户费** transfer fee
(0.1 bp, both sides) — see `CostModel`.

:::tip[Why the numbers drop]
Turnover-heavy strategies look *worse* under these rules than under the vectorized
engine — that's the point. T+1 and the ¥5 floor punish churn, and limit/suspension
guards remove fills you could never have gotten. Treat the gap as the realism tax.
:::

## Daily report

`daily_report(broker, fills, prices)` renders a Markdown summary (equity, fills,
costs, open positions, unrealized P&L) — generate it at close from a scheduled job.

## Going live on a QMT simulation account

1. On the **MiniQMT machine**, configure `QmtBroker(account_id=..., mini_qmt_path=...)`
   and verify the `xttrader` calls (scaffolded — must be checked on the box).
2. Run with `dry_run=True` and watch the logged orders.
3. Flip to `dry_run=False` with `QmtBroker` → orders reach your **simulation** account.

:::caution[Stop-loss can't be backtested by the vectorized engine]
The stop-loss is path-dependent, so the [vectorized backtest](./backtest) can't
validate it. The `SimBroker` here is the start of the event-driven path that can —
backtest the stop-loss before trusting it live.
:::
