# Execution ‚Äî `super_trade.execution`

The live-trading layer. It runs a strategy against a **broker**, with risk
controls, and is the bridge between backtest and live: the **same**
`Strategy` drives a simulated broker (dry-run / backtest) or a QMT broker (live on
a **simulation account**).

:::danger[Sensitive layer]
This layer can place orders. It defaults to **`dry_run=True`** (logs orders,
sends nothing), and is intended for a **QMT Ê®°ÊãüË¥¶Êà∑ (simulation account)**. Keep the
`RiskManager` limits on, and only set `dry_run=False` once you've watched the
dry-run output.
:::

A `Broker` exposes `cash()`, `positions()`, and `place_order()`. `SimBroker` fills
against an in-memory `Account` (cash + positions in real Êâã/shares), reusing the
backtest `CostModel` so costs match. `QmtBroker` submits via `xtquant.xttrader`.

## The scan loop

`ExecutionEngine.scan_once()` is one sweep of the universe:

1. read recent bars per symbol ‚Üí evaluate the strategy's target weight + price;
2. mark the account to market; apply the **daily-loss halt**;
3. **stop-loss exits first** (a name just stopped out is not re-bought this scan);
4. **signal entries/exits**, sized by the `RiskManager`;
5. submit orders through the broker (or just log them, in dry-run).

```python
from super_trade.execution import SimBroker, RiskManager, ExecutionEngine
from super_trade.backtest import SmaCross

engine = ExecutionEngine(
    broker=SimBroker(cash=1_000_000),
    store=store, strategy=SmaCross(10, 30), risk=RiskManager(),
    universe=["600519", "000001"], dry_run=True,
)
engine.scan_once()              # one pass
engine.run(interval_seconds=10) # scan every 10s (blocks; Ctrl-C to stop)
```

## Risk management

`RiskManager` is the **safety layer between a strategy's signal and an actual
order**. The strategy says *whether* to be long; the `RiskManager` decides *how much*
you may buy and *whether to block or force* a trade ‚Äî so risk rules live in one
place. It's used by the event-driven backtest and live execution (not the
vectorized engine).

### Limits (`RiskLimits`)

| Limit | Default | Meaning |
|---|---|---|
| `max_position_weight` | 0.20 | at most 20% of equity in any one name |
| `max_gross_exposure` | 0.95 | at most 95% invested in total (keep a cash buffer) |
| `max_daily_loss` | 0.05 | halt new buys once down 5% on the day |
| `stop_loss` | 0.08 | exit a name down 8% from its entry price |
| `max_orders_per_day` | 200 | cap on orders per session |

### What it does

- **Position sizing** ‚Äî `max_buy_shares(symbol, price, equity, broker)` returns the
  largest **board-lot-aligned** buy allowed: the **minimum** of three budgets, floored
  to whole Êâã (`shares = (budget // (price √ó 100)) √ó 100`):
  - *per-name*: `max_position_weight √ó equity` ‚àí what you already hold,
  - *gross*: `max_gross_exposure √ó equity` ‚àí total invested,
  - your available **cash**.
- **Stop-loss** ‚Äî `stop_triggered(pos, price)` is true when
  `price ‚â§ avg_price √ó (1 ‚àí stop_loss)`; the engine then sells that name.
- **Daily-loss halt** ‚Äî `start_day(equity)` snapshots the morning equity;
  `update_daily_loss(equity)` halts once equity drops past `max_daily_loss`.
- **Order-count cap** ‚Äî `can_trade()` becomes false after `max_orders_per_day`.
- **Kill switch** ‚Äî `kill()` halts everything immediately.

### Buys are gated; exits never are

When halted (daily-loss hit, kill switch, or order cap), **new buys are blocked but
sells and stop-losses always go through** ‚Äî you can always reduce risk. (`can_trade()`
is checked before entries; exits bypass it.)

### Per-session state

A `RiskManager` carries intraday state (the day's starting equity, order count, halt
flag), so **create a fresh one per backtest/session** and call `start_day()` at the
open ‚Äî the event-driven backtest and `ExecutionEngine` do this for you.

```python
from super_trade.execution import RiskManager, RiskLimits

risk = RiskManager(RiskLimits(max_position_weight=0.15, stop_loss=0.06))
```

## Daily report

`daily_report(broker, fills, prices)` renders a Markdown summary (equity, fills,
costs, open positions, unrealized P&L) ‚Äî generate it at close from a scheduled job.

## Going live on a QMT simulation account

1. On the **MiniQMT machine**, configure `QmtBroker(account_id=..., mini_qmt_path=...)`
   and verify the `xttrader` calls (scaffolded ‚Äî must be checked on the box).
2. Run with `dry_run=True` and watch the logged orders.
3. Flip to `dry_run=False` with `QmtBroker` ‚Üí orders reach your **simulation** account.

:::caution[Stop-loss can't be backtested by the vectorized engine]
The stop-loss is path-dependent, so the [vectorized backtest](./backtest) can't
validate it. The `SimBroker` here is the start of the event-driven path that can ‚Äî
backtest the stop-loss before trusting it live.
:::
